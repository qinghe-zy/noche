# noche 交互规则

## 1. 总原则

1. 写得顺，存得稳，找得到。
2. 草稿和正式内容严格分开。
3. 读写态明确分层。
4. 未来信是“封存”，不是普通保存。
5. 日历是辅助跳转页，不承担主导航。

## 2. recordDate 与保存规则

- `recordDate` 在“打开纸张”时锁定，不在保存时决定
- 空白内容不能正式保存
- 后台 / 失焦时必须静默暂存
- 暂存不等于正式保存，正式保存入口为“信封”

## 3. 草稿规则

- 草稿按类型隔离
- 日记草稿按日期分槽
- 随笔草稿第一版只保留一个活跃槽
- 未来信草稿单独分槽，不与随笔/日记混用

## 4. 未来信规则

- 日期最早只能选明天
- 解锁统一为目标日期本地时间 00:00
- 未解锁只显示锁定状态与解锁日期
- 封存后不允许编辑内容和日期
- 未来信草稿过期需要清空日期并提示重新选择

## 5. 阅读态与编辑态

- 阅读态与编辑态明确分层
- 阅读态右上角提供“续写”入口
- 不允许点击正文直接编辑

## 6. 信箱与排序规则

- “往日信件”按 `recordDate` 倒序
- 同一天内按 `createdAt` 倒序
- 严禁按 `updatedAt` 排序

## 7. 删除 / 销毁规则

- 删除统一走 `destroyEntry`
- 预留资源清理 Hook
- 删除后重新计算日历点
- 不做回收站

## 8. 日历规则

- 日历补写只能补日记
- 当天无记录进入今日日记
- 历史无记录只新建该日记
- 日历点仅表示日记/随笔内容存在

## 9. Deferred（暂挂项）

- App 跨天唤醒后的全局日期刷新策略
- 多内容日期点击是否进入列表页的最终体验
- 未来信解锁后在“往日信件”中的细粒度展示策略
- 后台任务缩略图隐私遮罩的最终实现
