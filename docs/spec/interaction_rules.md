# noche 交互规则

## 1. 总原则

1. 写得顺，存得稳，找得到。
2. 草稿和正式内容严格分开。
3. 读写态明确分层。
4. 未来信是“封存”，不是普通保存。
5. 日历是辅助跳转页，不承担主导航。
6. `diary / jotting / future` 在视觉层使用三套不同的编辑界面，但共享稳定的数据与保存语义。
7. 图片能力为本地优先、离线可用的最小能力，不依赖服务器。
8. diary 允许在写前先选择天气和心情，并在纸面顶部以缩略前序卡展示。

## 2. recordDate 与保存规则

- `recordDate` 在“打开纸张”时锁定，不在保存时决定
- 只有“正文为空且没有任何图片附件”时，才视为空白内容，不能正式保存
- diary 只有前序天气 / 心情、没有正文且没有图片时，仍视为空白内容
- 后台 / 失焦时必须静默暂存
- 暂存不等于正式保存，正式保存入口为“信封”

## 3. 草稿规则

- 草稿按类型隔离
- 日记草稿按日期分槽
- 随笔草稿第一版只保留一个活跃槽
- 未来信草稿单独分槽，不与随笔/日记混用
- 草稿保存时要连同图片附件一起保存与恢复
- diary 草稿保存时要连同 `diaryPrelude` 一起保存与恢复
- 历史日记草稿不能覆盖今日日记草稿

## 4. 图片附件规则

- `diary / jotting / future` 都允许插入本地图片
- 第一版只做本地图片插入、展示、删除、恢复
- 正式保存与阅读态都要保留图片引用
- 图片不塞进纯文本字段，使用独立附件模型
- 未解锁未来信不显示正文，也不显示图片全文
- 运行时不得依赖外链字体、CDN 样式、远程图片或外链 SVG

## 5. 未来信规则

- 日期最早只能选明天
- 解锁统一为目标日期本地时间 00:00
- 未解锁只显示锁定状态与解锁日期
- 封存后不允许编辑内容、图片和日期
- 未来信草稿过期需要清空日期并提示重新选择

## 6. 阅读态与编辑态

- 阅读态与编辑态明确分层
- 阅读态右上角提供“续写”入口
- 不允许点击正文直接编辑
- `EditorPage` 继续承担统一路由入口，但 diary / jotting / future 分别切到各自 shell
- diary 编辑态的前序卡允许轻点重新修改；阅读态默认只展示

## 7. 信箱与排序规则

- “往日信件”按 `recordDate` 倒序
- 同一天内按 `createdAt` 倒序
- 严禁按 `updatedAt` 排序

## 8. 删除 / 销毁规则

- 删除统一走 `destroyEntry`
- 预留资源清理 Hook
- 删除后重新计算日历点
- 不做回收站
- 若仅剩图片没有正文，仍允许保留，不自动判空销毁

## 9. 日历规则

- 日历补写只能补日记
- 当天无记录进入今日日记
- 历史无记录只新建该日记
- 日历点仅表示日记/随笔内容存在

## 10. Deferred（暂挂项）

- App 跨天唤醒后的全局日期刷新策略
- 多内容日期点击是否进入列表页的最终体验
- 未来信解锁后在“往日信件”中的细粒度展示策略
- 后台任务缩略图隐私遮罩的最终实现
- 图片附件的缩略图压缩、批量排序与更精细的资源清理策略
