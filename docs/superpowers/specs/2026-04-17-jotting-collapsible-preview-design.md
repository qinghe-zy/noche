# 随笔页预览态可折叠顶栏设计稿

## 1. 背景

当前随笔页在 `JottingEditorShell.vue` 中采用“两层固定 + 卡片内滚动”的结构：

- 页面最上方固定一层返回 / 完成按钮
- 卡片内部再固定一层大日期、标题、图片按钮与附件区
- 正文通过卡片内部的 `scroll-view` 单独滚动

这一结构会带来两个直接问题：

1. 顶部留白较大，返回与完成按钮下方到正文之间存在一段被固定头部占住、但未被有效利用的空间
2. 当正文内容未撑满当前卡片可视区时，预览区仍可继续上推，造成空推、松散和不稳定的阅读手感

本次改动目标是只针对随笔页预览态做一次结构性优化，在不推翻现有编辑器主流程的前提下，把随笔页改造成“内容驱动滚动 + 滚动折叠顶栏”的一体化体验。

## 2. 本次目标

### 2.1 体验目标

- 默认进入随笔页时，保持当前安静、舒展的大版式首屏
- 当内容不足以撑满正文预览区时，页面尽量保持不可滚动，不触发空推
- 只有当正文真实超出预览区后，页面才允许向上滚动
- 上滚后优先折叠头部，再滚正文
- 折叠完成后释放更多正文可视面积，改善当前固定死顶栏造成的僵硬感
- 编辑态保持现有输入逻辑，不强制回弹到展开首屏

### 2.2 范围

本次设计仅覆盖：

- 随笔页 `JottingEditorShell.vue`
- 随笔页在 `edit` 与 `read` 两个外部模式下的预览态和只读态交互
- 预览态滚动、折叠、进入编辑、双击修改、首次提示的交互口径

本次不覆盖：

- 日记页实现改造
- 未来信页改造
- `EditorPage.vue` 的全局模式重构
- 多页级路由变更
- 公共编辑框架的大重构

## 3. 已确认产品口径

### 3.1 随笔页核心口径

- 本次采用方案 A：在现有壳组件内扩展“预览滚动阶段”，不新增独立页面模式
- 本次改动既覆盖 `edit` 模式下的预览首屏，也覆盖 `read` 模式下的只读浏览
- 用户在预览中点击正文进入编辑时，保持当前位置接管输入，不强制回到展开态
- 折叠完成后的单行视觉形态为：

`[ 4月17日 · 这里是单行截断的标题... ] ------------------ [ 图片按钮 ]`

- 图片附件属于正文内容区，不属于可折叠头部
- 编辑预览态折叠完成后，返回、日期标题信息和右侧图片按钮合并为同一个顶部系统
- 只读 `read` 场景下折叠完成后，右侧不保留动作
- 折叠后的紧凑顶栏视觉为“介于卡片与系统导航之间”，保留轻纸面感，但弱化整块卡片边界

### 3.2 只读转编辑口径

- `read` 场景不再依赖顶栏“继续写”按钮
- 随笔页只读态改为“双击正文进入修改”
- 首次进入只读页时，给一次轻提示，例如“轻触阅读，双击继续写”
- 该提示不常驻，不做长期占位

### 3.3 滚动触发硬门槛

这是本次设计的硬门槛：

- 如果正文真实内容高度未达到滚动条件，则不得触发任何折叠相关机制
- 未达到滚动条件时，页面必须保持展开态
- 未达到滚动条件时，不得因为轻微上推手势产生顶栏收缩、重排、压缩等视觉变化

换句话说：

**折叠、压缩、重排这些机制只能在“内容已满足滚动条件”的前提下，被滚动触发。**

## 4. 状态设计

外部页面模式保持不变，仍由 `EditorPage.vue` 提供：

- `edit`
- `read`

随笔壳内部新增派生状态：

- `preview-expanded`
- `preview-collapsing`
- `preview-collapsed`
- `editing-inline`

### 4.1 `preview-expanded`

默认进入页面时的首屏状态：

- 展示大日期
- 展示标题输入框或标题文本
- 编辑预览态右侧展示图片按钮
- 头部保持大版式与舒展留白
- 内容不足时停留在该状态，不允许进入折叠

### 4.2 `preview-collapsing`

当且仅当正文真实内容高度超过预览区，并且用户开始上滚后进入：

- 滚动位移优先用于头部折叠
- 返回、标题、图片按钮逐步向紧凑顶栏重排
- 正文尚未主导滚动

### 4.3 `preview-collapsed`

折叠完成后的稳定状态：

- 顶部收束为单行紧凑顶栏
- 编辑预览态保留右侧图片按钮
- 只读态去掉右侧动作，保持安静
- 后续滚动主要交给正文内容

### 4.4 `editing-inline`

用户从预览进入输入后的状态：

- 不回弹
- 不强制展开头部
- 直接在当前位置接管 textarea 焦点与键盘
- 若输入行被键盘遮挡，只做最小必要补偿

## 5. 结构重组方案

### 5.1 当前结构问题

当前 `JottingEditorShell.vue` 的主要结构为：

- 最外层固定按钮层
- 卡片内部固定头部层
- 卡片内独立 `scroll-view`

这一结构会天然造成“头部先占坑，正文独立滚”的体验。

### 5.2 目标结构

目标是把随笔页改成“统一预览滚动 + 就地进入编辑”的单壳结构。

结构语义调整为：

- 页面最外层只保留页面容器
- 返回、日期、标题、图片按钮共同属于同一个折叠顶部系统
- 附件与正文一起下沉到内容区
- 预览阶段使用统一滚动语义
- 输入阶段由现有 textarea 光标和键盘联动逻辑接管

### 5.3 头部两种视觉形态

头部共用一份信息源，但有两种表现：

- 展开头部
  - 大日期
  - 标题输入框或标题文本
  - 编辑预览态右侧图片按钮
- 折叠头部
  - 左侧返回
  - 中间 `[日期 · 单行标题]`
  - 编辑预览态右侧图片按钮
  - 只读态无右侧动作

这不是两个完全独立页面，而是同一头部系统在滚动中的渐进压缩和重排。

### 5.4 附件归属调整

附件必须从“头部固定层”调整为“正文内容区”的一部分。

原因：

- 用户已明确要求图片附件与正文一起决定是否允许滚动
- 若附件继续留在头部层，会错误抬高头部高度，破坏“内容驱动滚动”的判断

因此本次结构上必须满足：

- 大日期 / 标题 / 图片动作属于可折叠头部
- 附件列表 + 正文文本属于内容区

## 6. 滚动与折叠规则

### 6.1 内容驱动滚动判定

先定义：

- `headerExpandedHeight`：展开头部高度
- `headerCollapsedHeight`：折叠头部高度
- `contentNaturalHeight`：正文内容区自然高度，包含附件、正文文本、必要下边距
- `contentViewportHeight`：展开态下正文可视区高度

判定规则：

- 若 `contentNaturalHeight <= contentViewportHeight`
  - 页面停留在 `preview-expanded`
  - 禁止触发折叠相关机制
- 若 `contentNaturalHeight > contentViewportHeight`
  - 页面允许进入可滚动系统
  - 后续滚动可驱动头部折叠

### 6.2 两段式滚动

定义：

- `collapseDistance = headerExpandedHeight - headerCollapsedHeight`

滚动位移消耗顺序：

1. 当滚动位移在 `0 ~ collapseDistance` 区间时，位移优先用于头部折叠
2. 当滚动位移超过 `collapseDistance` 后，额外位移才主要用于正文滚动

因此用户手感应表现为：

- 先收头部
- 再滚正文

### 6.3 折叠进度

统一采用线性进度：

- `collapseProgress = clamp(scrollOffset / collapseDistance, 0, 1)`

所有头部变化都绑定这一进度：

- 大日期字号缩小
- 标题由展开态压缩为单行截断
- 左右边距收紧
- 按钮对齐到折叠顶栏
- 展开头部透明度降低
- 折叠头部透明度升高
- 轻分隔线逐步显现

### 6.4 折叠完成的稳定性

为避免边界抖动：

- 当 `collapseProgress >= 1` 时，进入 `preview-collapsed`
- 当用户回拉时，只有当进度回退到较低阈值，例如 `< 0.92`，才退出折叠完成态

需要轻滞后区，避免边界附近反复跳变。

## 7. 动画口径

### 7.1 允许的动画手段

优先使用：

- `transform`
- `opacity`
- `z-index`
- 少量受控的 `padding / width / font-size`
- 少量状态驱动 JS

### 7.2 禁止的动画倾向

本次避免：

- 弹簧感
- 回弹感
- 3D 翻折
- 复杂物理模拟
- 高强度悬浮或机械感动势

目标气质应是：

“像纸面内容被轻轻收起，而不是组件做表演式弹跳。”

## 8. 进入编辑与只读修改规则

### 8.1 编辑模式预览态

在 `edit` 外部模式下：

- 页面先落在预览态，而不是直接把键盘逻辑推到最前
- 用户点击正文后，进入 `editing-inline`
- 保持当前滚动上下文，不回弹到大首屏

### 8.2 只读模式

在 `read` 外部模式下：

- 单击正文只阅读
- 双击正文进入修改
- 顶栏不保留“继续写”按钮

### 8.3 首次提示

只读页首次进入时：

- 在底部给一次轻提示
- 提示只出现一次
- 不常驻
- 不占用顶栏空间

## 9. 验收口径

### 9.1 核心验收

- 默认进入随笔页时保留大版式首屏观感
- 内容未达到滚动条件时，页面不得触发折叠相关机制
- 内容未达到滚动条件时，不得出现明显空推
- 内容达到滚动条件后，必须先折叠头部，再滚正文
- 折叠完成后的紧凑顶栏应符合：
  - 编辑预览态：`[返回] [日期 · 单行标题] [图片按钮]`
  - 只读态：`[返回] [日期 · 单行标题]`
- 附件必须属于内容区，与正文共同决定是否可滚动
- 从预览进入编辑时不得强制回到展开首屏
- 只读态通过双击正文进入修改
- 首次提示只出现一次，不常驻

### 9.2 非目标

本次不要求：

- 把随笔页和日记页一次性抽成通用公共编辑框架
- 修改一级路由结构
- 重做保存、草稿、附件预览等主流程

## 10. 实现边界与风险

### 10.1 实现边界

- 主改动集中在 `src/features/editor/components/JottingEditorShell.vue`
- `EditorPage.vue` 只允许做少量事件和状态承接，不做页面模式重构
- 公共键盘视口逻辑优先复用，不重新定义其交互语义

### 10.2 风险

- 预览滚动阶段与输入阶段的切换若处理不好，可能与现有键盘补偿逻辑打架
- 附件下沉到内容区后，需要重新校准内容高度测量
- 只读态双击进入修改会改变原有“继续写”入口，需要确保手势识别不干扰附件预览

### 10.3 已确认的实现暗坑

#### 暗坑一：图片附件导致的动态高度陷阱

本次滚动门槛依赖：

- `contentNaturalHeight > contentViewportHeight`

如果正文包含图片，而高度测量发生在图片尚未完成加载、尚未稳定占位的时间点，则会出现以下错误：

- 自然高度被低估
- 页面本应允许滚动，却被错误锁死在展开态
- 折叠顶栏完全不触发，形成“明明内容很多却推不动”的错误体验

因此实现时必须满足至少一条：

1. 内容高度只在 DOM 稳定且媒体资源加载完成后才作为最终判定依据
2. 图片附件在结构层有稳定占位，例如固定比例盒子，确保未完成加载时也能参与正确高度测量

推荐口径：

- 附件卡片在布局上使用稳定的宽高比占位
- 图片加载完成、附件列表变化、正文文本变化时，都触发内容高度重测
- 在高度未稳定前，不把第一次测得的偏小高度当作最终滚动门槛

#### 暗坑二：就地接管时的像素级跳动

`editing-inline` 要求：

- 不回弹
- 不重置滚动位置
- 在当前位置把只读文本或预览文本切换为激活态 `<textarea>`

这里存在一个高风险问题：

- 普通文本节点与原生 `<textarea>` 的排版引擎不同
- `<textarea>` 往往带有默认内边距、默认行盒表现和输入态差异
- 即便没有滚动位移，文字本身也可能在切换瞬间发生轻微闪烁、上下跳动或横向偏移

这会直接破坏“无缝接管”的目标。

因此实现时必须增加像素对齐要求：

- 只读文本与编辑 `<textarea>` 的字体、字号、行高、字距、颜色、内边距必须严格一致
- 明确关闭或覆盖原生输入框默认 padding 与默认外观
- 进入 `editing-inline` 时优先复用同一内容容器的宽度与排版上下文，避免因容器切换造成二次偏移
- 需要把“切换瞬间无明显文字跳动”列为单独的人工回归项

实现判断标准：

- 若用户在同一滚动位置进入编辑，正文第一屏的首行文字不应出现肉眼可见的抖动或重排感

## 11. 未来日记页扩展口径

本次不实现日记页，但设计时允许未来接入。

已确认的未来日记页折叠终态口径为：

- 折叠后保留一个很轻的 `Prelude` 标记或图标，不保留完整文案
- 主信息形态为：

`[ 4月17日 · 标题... · 天气/心情图标 ]`

- 日记 `read` 场景下折叠后右侧不保留动作
- `继续写` 改为与随笔一致的“双击正文修改”
- 首次进入只读页时同样给一次轻提示

该扩展口径仅作为后续兼容目标，不纳入本次实现范围。
