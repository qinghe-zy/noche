# noche architecture final

## 1. 文档目标

本文档用于确定 `noche` 在 Uni-app CLI + Vue 3 + TypeScript + Pinia 技术栈下的最终基础架构方向，作为后续功能开发、模块拆分和工程演进的统一依据。

## 2. 当前约束

- 当前阶段仅做最小可运行工程初始化。
- 不实现具体业务页面与业务流程。
- 不接入 SQLite 与本地持久化。
- 不引入额外 UI 框架。
- 保留现有分层目录骨架，后续在既定目录内扩展。

## 3. 技术栈结论

- 框架：`uni-app CLI`
- 视图层：`Vue 3`
- 语言：`TypeScript`
- 状态管理：`Pinia`
- 时间处理：`dayjs`
- 标识生成：`uuid`
- 构建方式：`Vite`

## 4. 分层设计

### 4.1 `src/app`

负责应用启动装配、全局注册和基础 provider 接入。

- `store/`：放置 Pinia 根级 store、store 插件和全局状态入口。
- `providers/`：放置后续全局能力提供者，例如主题、运行时配置、日志、同步状态等。

### 4.2 `src/domain`

负责核心业务概念与领域规则，不直接依赖页面实现。

- `entry/`：条目实体与规则。
- `draft/`：草稿实体与编辑中状态规则。
- `time/`：日期、时间段、日历相关领域对象与规则。
- `services/`：领域服务，负责跨实体规则组合。

### 4.3 `src/data`

负责数据访问抽象，不承载 UI 逻辑。

- `db/`：后续 SQLite 连接、表结构、迁移与事务封装。
- `repositories/`：仓储接口与实现。
- `mappers/`：数据库记录、领域对象、页面模型之间的转换逻辑。

### 4.4 `src/features`

负责按功能组织页面、组件、组合式逻辑。

- `home/`：首页能力。
- `editor/`：编辑器相关能力。
- `mailbox/`：收件箱相关能力。
- `calendar/`：日历相关能力。
- `profile/`：个人设置与账户相关能力。

原则：

- 页面、组件、组合逻辑优先内聚在 feature 内。
- Feature 之间不直接相互依赖页面实现。
- 跨 feature 共用逻辑应上移到 `shared` 或 `domain`。

### 4.5 `src/shared`

负责跨模块复用的通用能力。

- `ui/`：基础展示组件。
- `utils/`：纯函数工具。
- `constants/`：常量、枚举映射。
- `types/`：跨模块共享类型。
- `assets/`：通用静态资源。

## 5. 依赖方向

建议遵循以下依赖方向：

`features -> domain -> data`

补充说明：

- `features` 可以使用 `shared` 与 `app` 暴露的公共能力。
- `domain` 不依赖具体页面与组件。
- `data` 不反向依赖 `features`。
- `app` 只负责装配，不承载具体业务规则。

## 6. 初始化阶段的最小运行要求

初始化完成后，应具备以下能力：

- 可以执行依赖安装。
- 可以执行基础开发命令。
- 应用入口可正常启动。
- 至少存在一个最小占位页面用于验证工程可运行。
- Pinia 已完成注册，但暂不创建业务 store。

## 7. 后续演进顺序建议

1. 完成最小工程初始化与命令验证。
2. 建立全局别名、类型基线和规范约束。
3. 接入 Pinia 全局状态入口。
4. 增加 feature 级页面入口。
5. 接入 SQLite 数据层与仓储层。
6. 再逐步实现具体业务能力。

## 8. 当前明确不包含的内容

- 业务页面细节
- SQLite 表与迁移代码
- 网络同步逻辑
- UI 风格与主题细节
- 权限、登录、通知等扩展模块
