# noche interaction rules final

## 1. 文档目标

本文档定义 `noche` 的基础交互规则，目的是在业务实现开始前先统一“页面如何组织、状态如何切换、编辑如何处理”的规则边界。

## 2. 当前适用范围

- 适用于首页、编辑器、收件箱、日历、个人中心等后续模块。
- 当前阶段只定义规则，不实现具体页面。

## 3. 页面级交互规则

### 3.1 最小页面原则

- 每个页面在初期只承担单一主目标。
- 页面需要有可识别的主状态：加载中、空状态、可操作状态、错误状态。
- 不在初始化阶段叠加复杂弹层、复杂手势和动画。

### 3.2 导航规则

- 页面导航由 `pages.json` 统一声明。
- 页面路径采用稳定命名，避免后续频繁迁移。
- 页面切换优先保证状态明确，其次再优化转场体验。

## 4. 编辑类交互规则

### 4.1 草稿优先

- 所有编辑型页面优先操作 `Draft`，不要直接修改正式条目。
- 未保存内容应通过草稿状态体现。
- 保存动作与放弃动作需要在交互语义上明确区分。

### 4.2 变更提示

- 存在未保存修改时，页面离开前应具备提示机制。
- 自动保存能力可以后续增加，但初始化阶段不强制实现。

## 5. 时间类交互规则

- 所有日期时间展示统一通过时间工具层处理。
- 相同时间概念必须使用一致格式与时区解释。
- 日历视图与列表视图展示同一实体时，时间语义必须一致。

## 6. 状态管理规则

- 全局状态只放跨页面共享且有生命周期管理价值的数据。
- 局部页面状态优先留在页面或 feature composable 内。
- 不允许把纯展示状态无差别堆进全局 store。

## 7. 错误与空状态规则

- 空数据不等于错误。
- 错误状态必须能说明“出了什么问题”和“用户下一步可以做什么”。
- 初始化阶段允许使用极简错误提示，但结构上要预留扩展空间。

## 8. 可扩展性规则

- 组件命名与页面命名要反映职责，不用临时性命名。
- Feature 内部优先内聚，跨模块复用再抽离。
- 新增业务前先判断归属 `feature`、`domain`、`data`、`shared` 哪一层。

## 9. 当前明确未覆盖的内容

- 视觉规范与动效规范
- 多端差异化交互细则
- 离线同步冲突处理
- 通知、提醒、系统权限等复杂交互
