# noche 功能矩阵与交互逻辑（给 Codex 的执行文档）

## 1. 文档用途

本文件用于让 Codex 在接手 `noche` 项目时，快速理解：

1. 当前版本必须实现的功能范围
2. 每个页面与模块各自承载什么能力
3. 关键交互链路如何运行
4. 哪些能力属于 P0 / P1 / P2
5. 后续实现时哪些规则不能被改写

本文件不是宣传文案，也不是产品介绍，而是 **Codex 的执行说明书**。

---

## 2. 项目一句话定义

`noche` 是一个 Android-only、local-first 的私人写信式记录 App。

核心内容模型只有三类：
- 日记 diary
- 随笔 jotting
- 未来信 future

核心页面只有五类：
- Home
- Editor
- Mailbox
- Calendar
- Profile

产品重点不是复杂功能，而是：
- 写得顺
- 存得稳
- 找得到
- 有轻仪式感
- 页面安静克制，不像内容平台，也不像营销页

---

## 3. 执行总原则

Codex 后续实现必须遵守以下原则：

1. **规则先于页面**
   - 业务规则放在 store / domain / facade / repository，不要散落在页面里。

2. **前端实现要参考 `docs/stitch/**`**
   - 但不能保留原型文案、假数据腔、占位文字。

3. **不要推翻已有闭环**
   - 已跑通的 editor 最小闭环、settings persistence、bootstrap runtime、store facade 收口，应继续沿用和补强，不要返工重写。

4. **优先补主链路，不优先堆外围功能**
   - 当前最重要的是：写入 -> 保存 -> 阅读 -> 信箱查看 -> 日历跳转 -> 续写恢复。

5. **空白内容不能污染数据层**
   - 所有正式保存都必须经过空白内容拦截。

6. **不做当前版本明确排除的功能**
   - 不做复杂富文本、不做音频正式能力、不做云同步、不做企业后端、不做回收站、不做未来信封存后编辑。

## 5. 页面功能矩阵

## 5.1 Home

| 功能 | 是否必须 | 说明 |
|---|---:|---|
| 打开今日信纸 | 是 | 产品主入口 |
| 未来信入口 | 是 | 进入 future 编辑模式 |
| 邮箱入口 | 是 | 进入 Mailbox |
| 头像进入个人中心 | 是 | 进入 Profile |
| recent feed / dashboard | 否 | 当前版本不做 |

**页面职责**
- Home 是入口场，不是信息流，不是 dashboard。
- 首屏重点是：尽快进入写作或归档。

**页面禁止项**
- 不要堆大量卡片
- 不要放内容 feed
- 不要做营销式首屏

---

## 5.2 Editor

| 功能 | 是否必须 | 说明 |
|---|---:|---|
| 日记模式 | 是 | diary 编辑态 / 阅读态，视觉以 `docs/stitch/diary_editor` 为准 |
| 随笔模式 | 是 | jotting 编辑态 / 阅读态，视觉以 `docs/stitch/minimalist_jotting_editor` 为准 |
| 未来信模式 | 是 | future 编辑态 / 阅读态 / 锁定态，沿用信纸式设计 |
| 编辑态 | 是 | 输入内容与暂存 |
| 阅读态 | 是 | 正式保存后进入 |
| 自动静默暂存 | 是 | debounce + 退后台 |
| 信封正式保存 | 是 | 正式保存唯一入口 |
| 本地图片插入 | 是 | 三种模式都支持本机图片插入与删除 |
| future 日期选择 | 是 | 未选日期时弹出 |
| 续写 | 是 | 阅读态进入编辑态 |
| 销毁 | 是 | 阅读态可触发 |

**页面职责**
- 继续承接 diary / jotting / future 的写作、保存、阅读、续写。
- 继续保留统一路由入口，但内部按类型切换三套独立 editor shell。
- 这是灵魂页，但当前版本先保证闭环，再做细节美化。

**页面禁止项**
- 不允许点正文直接编辑
- 不允许自动暂存直接变正式保存
- 不允许 future 封存后继续编辑
- 不允许 diary / jotting 再套用 future 的同款视觉壳

## 7. 完整交互逻辑

## 7.1 今日日记主流程

### 入口
- Home 中间主卡：打开今日信纸

### 流程
1. 用户进入 Home
2. 点击“打开今日信纸”
3. 系统判断今天状态：
   - 若有未正式保存的今日日记草稿 -> 恢复草稿
   - 若有正式日记 -> 打开今天这篇，进入阅读态
   - 若都没有 -> 新建今日日记草稿
4. 进入 Editor
5. 用户输入内容
6. 系统静默暂存
7. 用户点击右上角信封
8. 若内容非空 -> 正式保存
9. 页面切换到阅读态
10. 轻反馈：收好了
11. 用户点击“续写” -> 回到编辑态
12. 修改后再次点信封 -> 回到阅读态

### 不能改的规则
- 日记按天唯一
- 正式保存入口只能是信封
- `recordDate` 在打开纸张时锁定

---

## 7.2 随笔流程

### 入口
- 独立“写随笔”入口，不与今日日记混用

### 流程
1. 用户点击“写随笔”
2. 系统检查是否已有活跃随笔草稿
3. 若无 -> 新建随笔草稿
4. 若有 -> 弹出冲突选择：
   - 继续上次
   - 另起一张
   - 取消
5. 若选择“另起一张”且旧草稿已有内容：
   - 先将旧随笔正式收好
   - 再打开一张新的随笔
6. 若选择“另起一张”但旧草稿为空：
   - 安静收起旧草稿
   - 再打开一张新的随笔
7. 用户进入编辑态
8. 输入内容
9. 系统静默暂存
10. 用户点击信封
11. 若内容非空 -> 正式保存为一条新的随笔
12. 进入阅读态
13. 内容出现在“往日信件”中，并带“随笔”标签

### 不能改的规则
- 随笔不与今日日记共用内容
- 正式随笔数量不限
- 活跃随笔草稿第一版只能有一个

---

## 7.3 未来信流程

### 入口
- Home 下方轻入口：未来信

### 流程
1. 用户点击“未来信”
2. 系统检查是否已有活跃 future 草稿
3. 有 -> 恢复草稿
4. 无 -> 新建 future 草稿
5. 进入 future 编辑模式
6. 用户输入正文
7. 点击右上角信封
8. 系统检查是否已选日期：
   - 未选 -> 弹出底部半屏日期选择器
   - 已选 -> 直接进入封存
9. 日期选择器只允许选择明天及以后
10. 用户确认日期
11. 系统正式封存
12. 轻反馈：已封存
13. 内容进入“待启之信”
14. 到达目标日期本地时间 00:00
15. 自动转为可读状态
16. 从“待启之信”移入“往日信件”
17. 用户可进入阅读态查看全文

### 不能改的规则
- future 最早只能选明天
- 封存后不可编辑、不可改日期
- 未解锁时不能看全文

---

## 7.4 信箱浏览流程

### 入口
- Home 下方轻入口：邮箱

### 流程
1. 用户进入 Mailbox
2. 默认展示两个 tab：
   - 往日信件
   - 待启之信
3. 在“往日信件”中查看：
   - diary
   - jotting
   - 已解锁 future
4. 在“待启之信”中查看：
   - 未解锁 future
5. 用户点击内容卡片：
   - 已保存 diary / jotting -> 进入阅读态
   - 已解锁 future -> 进入阅读态
   - 未解锁 future -> 只显示开启日期与锁定状态
6. 点击右上角日历按钮 -> 进入 Calendar

### 不能改的规则
- 往日信件排序：`recordDate` 倒序，同日按 `createdAt` 倒序
- 不按 `updatedAt` 排序

---

## 7.5 日历跳转流程

### 入口
- Mailbox 右上角日历图标

### 流程
1. 用户进入 Calendar
2. 页面展示月历
3. 有记录的日期带小圆点
4. 用户点击某日期：
   - 若当天只有一条内容 -> 直接进入该内容阅读态
   - 若当天有多条内容 -> 进入当天归档列表
   - 若当天无记录且是历史日期 -> 新建该日期 diary
   - 若当天无记录且是今天 -> 进入今日日记新建/恢复逻辑
5. 点击右上角“回到今天” -> 回到当前月份并高亮今天

### 不能改的规则
- 日历只允许补写 diary
- 不允许通过 Calendar 补写 jotting

---

## 7.6 当天归档列表流程

### 触发条件
- 用户点击 Calendar 中某个有多条内容的日期

### 流程
1. 进入当天归档列表页
2. 列出该日期下所有内容：
   - diary
   - jotting
   - 已解锁 future（若纳入）
3. 用户点击某一条
4. 进入对应内容阅读态

---

## 7.7 销毁流程

### 入口
- 阅读态底部轻文字“销毁”
- 或长按纸面触发

### 流程
1. 用户触发销毁
2. 系统弹确认
3. 用户确认后执行 `destroyEntry()`
4. 删除记录
5. 刷新 Mailbox 列表
6. 刷新 Calendar 标记
7. 若该日期下已无任何 diary 或 jotting，则日历点消失

### 特殊情况
- 已解锁 future 可以销毁
- 未解锁 future 也可以销毁

---

## 7.8 后台恢复流程

### 流程
1. 用户正在输入内容
2. App 退到后台
3. 系统立刻静默暂存
4. 若开启隐私锁，最近任务缩略图不显示正文
5. 用户回到前台
6. 若开启隐私锁，先显示解锁页
7. 解锁成功后恢复到离开前页面
8. 草稿内容、光标位置、future 日期都要恢复

---

## 7.9 空白内容处理流程

### 场景 A：未保存草稿为空
1. 用户进入编辑页
2. 没写内容，或删空
3. 点击信封
4. 系统判断 `content.trim()` 为空
5. 不生成正式记录
6. 静默收起
7. 可轻提示：信纸是空的，已收起

### 场景 B：已正式内容被删空
1. 用户打开一篇已有内容
2. 点击续写并删空正文
3. 点击信封
4. 系统弹确认：这封信已经空了，要销毁吗？
5. 用户确认后再销毁

---

## 8. 关键业务规则清单（Codex 不得改写）

1. 日记按天唯一
2. `recordDate` 在打开纸张时锁定，不在保存时决定
3. jotting 可无限正式保存，但第一版只有一个活跃草稿槽
4. future 最早只能选明天
5. future 封存后不可编辑，不可改日期
6. future 解锁时刻为目标日期本地时间 00:00
7. 未解锁 future 不能看全文
8. 正式保存唯一入口是信封
9. 自动静默暂存不等于正式保存
10. 空白内容不能正式保存
11. 删除统一走 `destroyEntry()`
12. Calendar 只能补写 diary
13. Mailbox 排序不能按 `updatedAt`
14. Profile 不是后台控制台
15. Home 不是 dashboard

## 9. 当前版本明确不做

以下能力当前版本不做，Codex 不要主动扩展：

- 复杂富文本
- 图片 / 音频正式功能
- 社交分享
- 多人协作
- 云同步
- 企业级后端
- 回收站
- 多个活跃随笔草稿并存
- future 封存后编辑
- 复杂统计分析
- 分钟级 future 倒计时
