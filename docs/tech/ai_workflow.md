# noche AI workflow

## 1. 角色分工

- `Antigravity`：负责任务编排、拆解、验收和阶段推进
- `Codex`：负责工程、底层、数据、配置、可编译骨架，以及当页面改动明显依赖 domain / data / storage / test 联动时的端到端实现
- `Gemini`：在已约定写入边界内负责页面与 UI 组件实现，可作为视觉加速器，而不是唯一页面实现方

## 2. 写入边界

- 同一时间、同一目录只允许一个 AI 写入
- 一个 AI 正在改动某个目录时，其他 AI 只能读，不能并发写
- 跨目录协作前要先明确负责范围，避免覆盖彼此改动

## 3. 推荐协作方式

### 3.1 Antigravity

- 产出任务拆解
- 指定目录负责人
- 在阶段结束时做验收

### 3.2 Codex

- 初始化工程
- 补齐配置
- 维护 domain / data / shared / app 骨架
- 处理依赖、构建、验证
- 当页面需求牵涉真相文档、数据模型、持久化或测试矩阵时，直接承担文档 + 数据 + UI 的一体化收口

### 3.3 Gemini

- 基于已稳定的接口与路由做页面实现
- 负责视觉层和交互层组件
- 不直接改底层 schema、store 骨架和工程配置

## 4. 当前协作规则

- 先由 Codex 把底层工程和可编译骨架稳定下来
- 页面若只涉及纯视觉或局部结构，可交由 Gemini 基于稳定接口实现
- 页面若同时改动 visual shell、draft/save 规则、附件模型或本地持久化，则优先由 Codex 统一收口
- Antigravity 负责验收规则是否与产品文档一致
- 每完成一个阶段需生成 handoff / checkpoint 文档
- 稳定阶段需本地 commit，防止上下文丢失
- 当前 Editor 真相已更新为三套视觉壳：
  - `DiaryEditorShell`
  - `JottingEditorShell`
  - `FutureLetterEditorShell`
- 当前图片能力真相已更新为：三种类型都支持本地图片插入、展示、删除、恢复，全程本地优先、离线可用

## 5. 禁止事项

- Gemini 不在未约定的情况下改动底层工程配置
- 多个 AI 不得同时在同一路径下提交写入
- 禁止将明文数据库密码写入将要提交的文件
- 不要把图片能力实现成依赖服务器的上传流
- 不要为了分三套 editor shell 推翻既有路由和保存主链
