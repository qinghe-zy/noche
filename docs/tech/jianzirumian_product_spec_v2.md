# 见字如面产品说明 v2

版本：v2  
状态：可进入开发前定稿  
适用阶段：Android 单端、本地优先、持续迭代

## 1. 产品定位

### 1.1 一句话定义
一个以“写给自己”为核心隐喻的私人记录 App，帮助用户写日记、写随笔、给未来写信，并通过信箱与日历回看自己的时间痕迹。

### 1.2 核心隐喻
- 写内容 = 写一封信给自己
- 点信封 = 把这张纸收起来 / 正式封存
- 回看内容 = 打开私人信箱
- 未来信 = 封存到指定日期再开启的信

### 1.3 当前平台范围
- 当前阶段只做：**Android 手机端**
- 当前阶段不做：Windows / 平板 / iOS / Web 正式版本

### 1.4 产品气质
- 极简
- 温暖
- 私人化
- 纸张感
- 信件感
- 安静、克制、有轻仪式感

## 2. 当前范围与非目标

### 2.1 当前核心能力
1. 写日记
2. 写随笔
3. 给未来写封信
4. 查看信箱
5. 按日历跳转记录

### 2.2 当前明确不做

- 不做多人协作

- 不做复杂提醒系统
- 不做图片音频混合编辑器
- 不做云端图片上传与账号同步

## 3. 内容模型与业务定义

### 3.1 三种核心内容类型

#### A. 日记 Diary
- 按天唯一
- 每个自然日最多 1 篇日记
- 可多次追加和修改
- 可通过日历补写历史日期
- 进入 diary 编辑前可先选择天气和心情
- 这组前序元信息只作用于 diary，不扩散到 jotting / future
- 天气和心情允许跳过，跳过后仍可正常写作
- 进入 diary 后，纸面顶部可展示缩略前序卡

#### B. 随笔 Jotting
- 随时记录灵感
- 一天可以有无数多条随笔
- 不支持通过日历补写历史随笔
- 允许存在一个“活跃随笔草稿槽”

#### C. 未来信 Future Letter
- 写给未来的自己
- 必须指定未来日期
- 封存后在指定日期本地时间 00:00 解锁
- 正式封存后不可编辑、不可改日期，只能等待开启或销毁

### 3.2 页面内展示口径
- 日记与随笔、已解锁未来信都属于“往日内容”
- 信箱标签页命名统一为：
  - **往日信件**
  - **待启之信**

> 说明：这里不再使用“往日随笔”作为总称，因为它无法完整承接“日记 + 随笔 + 已解锁未来信”。

## 4. 页面结构

### 4.1 首页 Home
作用：进入核心写作与归档空间。

入口：
- 中间主入口：打开今日信纸 / 写今日内容（沿用当前设计稿中心主卡）
- 下方入口：日记、未来信、邮箱
- 右上角头像：个人中心

说明：
- 首页是入口场，不是信息中心
- 不放底部导航
- 不放 Recent feed
- 不放悬浮按钮

### 4.2 通用编辑页 Editor
作用：继续承担统一路由入口与 orchestration，但内部按类型切到三套独立 editor shell。

模式：
- 日记模式
- 随笔模式
- 未来信模式
- 阅读态（只读）

### 4.3 信箱页 Mailbox
作用：回看过去内容与查看未开启未来信。

标签页：
- 往日信件
- 待启之信

入口：
- 首页点“邮箱”进入
- 右上角日历图标进入日历页

### 4.4 日历页 Calendar
作用：按日期查看是否有记录，并跳转到当日内容。

入口：
- 从信箱页右上角日历图标进入

退出：
- 左上返回回信箱页

右上角：
- 不再跳新页面
- 右上角图标定义为“回到今天”

### 4.5 个人中心 Profile
作用：个人信息、外观、隐私锁、备份、关于。

入口：
- 首页右上角头像

### 4.6 当天归档列表页 / 当天内容列表（辅助页）
作用：当用户从日历点进某个日期，而该日期有多条内容时，展示该日期下的内容列表。

触发条件：
- 某日期下存在多条内容（例如 1 篇日记 + 2 条随笔）

## 5. 核心交互流

### 5.1 首页
- 点中间主卡 → 进入当天写作入口
- 点“日记” → 进入日记相关入口 / 当日日记
- 点“未来信” → 进入未来信编辑模式
- 点“邮箱” → 进入信箱页
- 点头像 → 进入个人中心

### 5.2 编辑页
- 默认进入即聚焦写作
- 输入过程中自动静默暂存草稿
- 点右上角信封 = 正式收好 / 保存
- 正式保存后，当前页由编辑态切为阅读态
- 阅读态右上角提供“续写”入口，进入编辑态
- 不允许点正文直接进入编辑态
- diary 编辑前若当前草稿没有前序元信息，可先展示天气 / 心情选择层
- diary 编辑态与阅读态都可在顶部展示缩略前序卡

### 5.3 信箱页
- 在“往日信件”查看已保存内容
- 在“待启之信”查看未开启未来信
- 点内容卡片进入阅读态
- 点右上角日历图标进入日历页

### 5.4 日历页
- 有记录日期带点/线标记
- 点有记录日期：
  - 若该日期只有 1 条内容，直接进入该内容阅读态
  - 若该日期有多条内容，进入“当天归档列表页”
- 点无记录历史日期：固定新建该日期的**日记**
- 点无记录且是今天：进入今日日记新建/续写

### 5.5 未来信
- 进入未来信编辑模式
- 编写内容
- 点信封时，如果还没选日期，则弹出底部半屏日期选择器
- 选定日期后立刻封存
- 未到日期不可查看全文
- 到指定日期 00:00 自动解锁

## 6. 关键业务规则（整合版）

### 规则 1：日记与随笔分离
- 日记：按天唯一，可追加
- 随笔：不限数量，随时新建

### 规则 2：今日日记入口
- 点击“写日记 / 打开今日信纸”时：
  - 若当天已有日记，则打开当日日记
  - 若当天没有日记，则创建当日日记

### 规则 3：随笔正式记录数量
- 一天可保存无数多条随笔
- 不限制已保存随笔数量

### 规则 4：活跃随笔草稿数量
- 第一版同一时间只保留 1 个活跃随笔草稿槽
- 点击“新建随笔”时：
  - 若已有未封存随笔草稿，弹出底部选项：
    - 继续上次
    - 另起一张
    - 取消
- 选择“另起一张”时，若当前随笔已有内容，应先自动收好当前随笔，再进入新的空白随笔

### 规则 5：草稿自动暂存
- 输入过程中 debounce 2~3 秒静默暂存一次
- App 退后台时强制再暂存一次
- 暂存不等于正式保存

### 规则 6：正式保存入口
- 点右上角信封才算正式保存 / 正式封存
- 自动暂存只用于防丢，不用于生成正式记录

### 规则 7：阅读态与编辑态分离
- 正式保存后切到阅读态
- 阅读态右上角提供“续写”入口进入编辑态
- 不允许点正文直接编辑

### 规则 8：空白信纸不生成记录
- 点击信封时，如果 `content.trim()` 为空：
  - 若当前只是未正式保存草稿，且没有图片、也没有正文：静默丢弃，不生成记录，不打日历点
  - 若当前是已正式存在的内容被编辑到全空：弹出“这封信已经空了，要销毁吗？”

补充：
- diary 已选天气 / 心情，但没有正文且没有图片时，仍视为空白 diary
- 这类内容允许保留在草稿中，但不允许正式生成 entry

### 规则 9：删除
-
- 不使用垃圾桶图标
- 阅读态可通过底部小字删除或长按信纸触发
- 销毁后彻底物理删除，不做回收站
- 删除后重新计算日历标记
- 若某日期下已无任何日记/随笔，则日历点消失

### 规则 10：recordDate 锁定时机
- `recordDate` 在“打开这张纸”的瞬间锁定
- 不是在点击保存时决定
- 即使跨天保存，内容仍属于打开时那一天

### 规则 11：时间字段分离
每条记录至少有：
- `recordDate`：内容属于哪一天
- `createdAt`：首次创建时间
- `updatedAt`：最后编辑时间
- `savedAt`：正式封存时间（首次或最近一次正式保存）

### 规则 12：日历补写权限
- 日历点击历史空白日期，只能补写**日记**
- 不允许通过日历补写随笔
- 若该日期已有日记，则直接打开那篇日记
- 若该日期只有随笔没有日记，仍可新建该日期日记

### 规则 13：信箱排序规则
- “往日信件”列表按 `recordDate` 倒序排列
- 同一天内多条内容再按 `createdAt` 倒序排列
- 严禁按 `updatedAt` 排序

### 规则 14：未来信日期选择时机
- 未来信模式下，日期不是前置必填
- 点信封时若未选日期，则弹出底部半屏日期选择器
- 选完日期后立刻执行封存

### 规则 15：未来信日期可选范围
- 未来信日期选择器禁选今天及更早日期
- 最小可选日期 = 明天
- 解锁时间统一为目标日期本地时间 00:00:00

### 规则 16：未来信草稿过期校验
- 恢复 `draft_future` 时，若 `unlockDate <= 今天`：
  - 清空原日期
  - 保留正文
  - 提示“原定日期已过，请重新选择开启时间”
- 用户必须重新选日期后才能封存

### 规则 17：未来信封存后权限
- 正式封存后的未来信：
  - 不可编辑内容
  - 不可修改解锁日期
  - 只允许等待开启或销毁

### 规则 18：未来信解锁行为
- 到达目标日期 00:00 后自动解锁
- 锁图标消失或变为开启状态
- 若用户第一次打开 App 时已过解锁时间，启动时自动刷新状态

### 规则 19：待启之信与已解锁未来信归属
- 未解锁未来信显示在“待启之信”
- 已解锁未来信归入“往日信件”

### 规则 20：草稿按类型隔离
草稿至少分为：
- `draft_diary_{recordDate}`
- `draft_jotting`
- `draft_future`

说明：
- 日记草稿必须按日期分槽
- 随笔草稿第一版只保留一个活跃槽
- 未来信草稿单独分槽，不能与随笔/日记混用
- diary 的天气 / 心情前序元信息随对应 `draft_diary_{recordDate}` 一起暂存与恢复

### 规则 21：历史日记编辑不覆盖今日日记草稿
- 编辑历史日记时，其草稿必须写入对应 `draft_diary_{recordDate}`
- 不允许覆盖今天的日记草稿

### 规则 22：隐私锁与草稿恢复
- App 退后台时先静默暂存当前草稿
- 若开启隐私锁，回到前台时先解锁
- 解锁成功后恢复到离开前页面与草稿状态
- 一个字都不能少

### 规则 23：最近任务缩略图隐私保护
- App 退后台后，最近任务缩略图不得显示正文
- 应显示模糊纸面或纯米白占位页

### 规则 24：多内容日期跳转
- 日历点击某日期时：
  - 若仅 1 条内容，直接进入
  - 若多条内容，进入“当天归档列表页”

### 规则 25：日历打点范围
- 日历点只表示该日期有“日记或随笔”内容
- 第一版未来信不单独参与日历打点逻辑
- 删除日记/随笔后，如该日期已无任何内容，则标记点消失

### 规则 26：日期与时区口径
- 第一版所有日期判断均按**当前设备本地日期**执行
- `unlockDate` 推荐存为 `YYYY-MM-DD`
- 允许用户手动改系统时间影响结果，不额外对抗

### 规则 27：标题自动生成
- 第一版不强制用户填写标题
- 列表标题自动取正文第一行前 12~16 个字生成
- 无正文则不生成正式记录

### 规则 28：删除实现方式
- 删除必须统一走 `destroyEntry(id)` 封装
- 不允许页面里直接裸删数据库记录
- `destroyEntry()` 职责：
  1. 查记录
  2. 清理关联资源（未来扩展点）
  3. 删数据库记录
  4. 触发列表与日历刷新

### 规则 29：暂不处理项（挂起）
- App 存活跨天后首页“今日日期状态”自动刷新问题（此前内部列为潜在问题 13）
- 当前版本先不写入正式实现规则，后续进入全局状态管理时再补

- 从信箱页右上角日历图标进入

### 7.2 右上角图标
- 定义为：回到今天
- 功能：

  - 高亮今天
  - 若今天有记录，则选中今天

- 如果是历史日期：新建该日期日记
- 如果是今天：新建/打开今日日记
- 不允许通过日历新建随笔

### 8.1 标签页
- 往日信件
- 待启之信

### 8.2 往日信件
包含：
- 日记
- 随笔

## 9. 草稿系统

### 9.1 草稿槽定义
- `draft_diary_YYYY-MM-DD`
- `draft_jotting`
- `draft_future`

### 9.2 草稿存储内容
每个草稿至少保存：
- `type`
- `content`
- `recordDate`
- `updatedAt`
- `cursorPosition`
- `unlockDate`（未来信）
- `mode`
- `diaryPrelude`（仅 diary 使用，可为空）

### 9.3 草稿恢复策略
- 点日记 → 恢复该日期日记草稿
- 点随笔 → 恢复活跃随笔草稿或弹冲突选择
- 点未来信 → 恢复未来信草稿，并做日期有效性校验

```ts
interface Entry {
  id: string
  type: 'diary' | 'jotting' | 'future'
```

## 11. 技术选型（当前建议版）

### 11.1 客户端技术栈
- 框架：Uni-app CLI
- 语言：TypeScript
- 视图层：Vue 3 + Composition API + `<script setup>`
- 状态管理：Pinia
- 日期处理：dayjs
- ID 方案：UUID

### 11.2 数据存储
- 核心内容仓库：SQLite
- 轻量配置：`uni.setStorageSync`
- 删除动作统一走 `destroyEntry()`
- 文件资源层：本地图片能力已经进入正式范围，继续保持本地引用与离线可用
- 天气 / 心情资源必须本地化，不能依赖外链字体、CDN 或远程图片

### 11.3 架构原则
- Android-only
- Local-first
- 页面只管展示
- `domain` 管业务规则
- `data` 管持久化
- 为未来加图片/音频/备份/同步预留演进空间

### 11.4 建议目录骨架（未来工程初始化时使用）
```text
src/
  app/
  domain/
  data/
  features/
  shared/
```

## 13. 非功能性要求

### 13.1 隐私
- 后台缩略图隐藏正文
- 隐私锁恢复后必须回到离开前状态

### 13.2 稳定性
- 草稿不能丢
- 历史编辑不能覆盖今天草稿
- 空白内容不能污染数据库
- 排序不能因更新旧内容而错乱
- 飞行模式下 diary 的天气 / 心情前序卡也必须可恢复、可展示

### 13.3 可扩展性
- 删除逻辑为未来附件预留清理 Hook
- 领域规则集中在 `domain` 层
- 内容类型与时间字段一开始就分离清楚

## 14. 当前设计与实现注意事项

1. 首页是入口场，不是 dashboard
2. 编辑页是灵魂页，要保留信纸感与阅读/编辑态切换
3. 信箱是档案盒，不是内容 feed
4. 日历是辅助跳转，不是主功能页
5. 个人中心是安静角落，不是系统后台
6. 任何实现不得反向改写已拍板的业务规则
